# 購入クォータ採用ガイド

ユーザー単位の購入金額に上限を設けるための仕組み。レートリミット (回数制御) と独立して動作する。

## ユースケース例

- 銀行振込 (インハウス) は 1 時間あたり 1 ユーザー 30 万円まで
- 決済方法に関わらず 1 ユーザー 1 日 100 万円まで
- マネーロンダリング対策・KYC ルール準拠
- 不正アクセス検知 (大量購入アラート)

## 有効化手順

### 1. 設定ファイルにルールを追加

`src/config/app/purchase-quota.config.ts` の `PURCHASE_QUOTA_RULES` 配列にルールを並べる。

```ts
export const PURCHASE_QUOTA_RULES: readonly QuotaRule[] = [
  {
    key: "bank_transfer_hourly",
    label: "銀行振込: 1時間30万円",
    scope: { type: "paymentMethod", value: "bank_transfer" },
    windowSeconds: 3600,
    maxAmount: 300_000,
  },
  {
    key: "all_daily",
    label: "全決済: 1日100万円",
    scope: { type: "all" },
    windowSeconds: 86400,
    maxAmount: 1_000_000,
  },
];
```

複数ルールが同時に評価され、いずれか 1 つでも超過すれば購入が拒否される。

### 2. DB マイグレーション

新規テーブル `purchase_quota_ledger` を作成する。

```bash
pnpm drizzle-kit push
```

既存テーブルは変更されないため、稼働中プロジェクトもダウンタイム無しで導入可能。

### 3. cron 登録

90 日より古い ledger 行を削除する cron を登録 (任意・推奨)。

```bash
pnpm task purchase-quota-cleanup
```

推奨スケジュール: `0 4 * * *`

## 動作確認

設定後、上限を超える購入を試みると以下のレスポンスが返る:

- HTTP 429
- メッセージ: `購入上限を超えました (<ルール label>)。しばらく経ってから再度お試しください。`

監査ログ (管理画面 `/admin/audit-logs`) で `purchase_quota.reservation.exceeded` action を検索すると、超過拒否イベントの履歴を確認できる。

## 無効化

`PURCHASE_QUOTA_RULES` を空配列 `[]` に戻すだけで完全に無効化される。
台帳テーブルはそのまま残るが、集計クエリは走らないため性能影響ゼロ。

## 公開 API

ドメイン詳細は `src/features/core/purchaseQuota/README.md` を参照。

```ts
import {
  reserveQuota,
  commitQuota,
  releaseQuota,
  getQuotaUsage,
  cleanupOldLedger,
  PurchaseQuotaExceededError,
} from "@/features/core/purchaseQuota/services/server/wrappers/purchaseQuotaHelper";
```

カスタム購入フロー (例: 管理者による手動付与) で台帳整合を保ちたい場合は、トランザクション内で `reserveQuota` / `commitQuota` / `releaseQuota` を直接呼び出すこと。

## レートリミットとの違い

| | rateLimit (既存) | purchaseQuota (本機能) |
|---|---|---|
| 制御対象 | リクエスト**回数** | 累積**金額** |
| 識別子 | IP / サブネット | userId |
| 上限の種類 | maxRequests | maxAmount (円) |
| 適用箇所 | routeFactory の `rateLimit` オプション | initiatePurchase 内 (固定組込) |
| ロールバック | 不要 | 失敗・キャンセル時に release |

2 つは独立して動作し、両方同時に有効化できる。
