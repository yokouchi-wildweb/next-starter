# 決済システムのプロバイダ追加方法

新しい決済プロバイダを追加する手順。

## 設計思想

- **プロバイダ抽象化**: `PaymentProvider` インターフェースにより実装を隠蔽
- **メソッド単位のプロバイダルーティング**: `paymentMethods[i].provider` をソースオブトゥルースとし、ユーザーが選んだ支払い方法から `resolveProviderForMethod()` で担当プロバイダを解決する（1メソッド=1プロバイダ）
- **起動方式の多態化**: `createSession` は `LaunchInstruction` 型（`redirect` / `client_sdk` のユニオン）を返し、クライアント側 `executePaymentLaunch` が type で分岐
- **Webhook 3分岐**: status フィールド（completed/failed/pending）で処理を制御
- **金額照合**: paidAmount でプロバイダ側の実課金額を検証
- **Webhook識別子リゾルバー**: プロバイダ固有のID体系に対応

### 起動方式の 2 系統

プロバイダは決済の起動方式によって 2 つに分類される。実装ファイル数と動線が異なるため、最初にどちらかを判定する。

| 系統 | 例 | createSession 戻り値 | ユーザー画面遷移 | 確定経路 |
|---|---|---|---|---|
| **redirect 型** | Stripe / Fincode / Square / inhouse | `{ type: "redirect", url: "https://..." }` | provider のホスト型ページに遷移 | success_url 戻り + Webhook |
| **client_sdk 型** | Paidy | `{ type: "client_sdk", sdkName, config }` | 同一ページ上のモーダル（SDK 起動） | クライアント確定 API + Webhook |

新規プロバイダがどちらに該当するかで追加するファイル数が変わる。本ドキュメントでは redirect 型を主軸に手順を示し、client_sdk 型固有の差分は別途明記する（[client_sdk 型の追加差分](#client_sdk-型の追加差分) 参照）。

## PaymentProvider インターフェース

```typescript
// src/features/core/purchaseRequest/types/payment.ts

interface PaymentProvider {
  readonly providerName: string;

  // 【必須】起動方式。redirect（URL遷移）/ client_sdk（ページ内SDK）。
  // resolveInitiation が「processing リクエストの再起動方法」を判定する網羅軸。
  // redirect 型は保存済み redirect_url を再利用、client_sdk 型は createSession を再実行する。
  readonly launchType: "redirect" | "client_sdk";

  // 【必須】Webhook/ステータス照会で purchase_request を逆引きする照合キーの種別。
  // "session_id"（payment_session_id・多数派の既定）/ "order_id"（provider_order_id・Fincode 等）。
  readonly correlationKey: "session_id" | "order_id";

  // 【任意】purchase_request.id から provider 固有の order_id を導出する。
  // 独自の order 識別子を持つ provider（Fincode/inhouse 等）が実装する。
  deriveProviderOrderId?(purchaseRequestId: string): string | undefined;

  // 同一 purchaseRequestId に対し冪等であること（resume が createSession を再実行するため）。
  // 外部副作用を持つ provider は idempotency key で同一セッション/Order を再利用する。
  createSession(params: CreatePaymentSessionParams): Promise<PaymentSession>;

  verifyWebhook(request: Request): Promise<PaymentResult>;
  verifyWebhookSignature?(request: Request, signature: string): Promise<boolean>;
  // paymentMethod は購入時にユーザーが選択した値（例: "credit_card"）。
  // プロバイダ側の照会 API がメソッド種別を要求する場合に使用する（Fincode の pay_type 等）。
  getPaymentStatus?(sessionId: string, paymentMethod?: string): Promise<PaymentStatusResult>;
}
```

> **冪等キーと購入開始フロー**: クライアント（`useCoinPurchase`）は冪等キーを購入 intent
> （支払い方法・ウォレット・金額）から発番し、方法/金額を切り替えると別キー＝別 purchase_request
> になる。サーバーは `findByIdempotencyKey(key, userId)`（**user スコープ必須**）で既存を引き、
> 純粋関数 `resolveInitiation` が `InitiationOutcome`（create / reuse-pending / resume /
> completed / conflict）を返す。`initiatePurchase` はそれを dispatch するだけの薄い層。
> 新 provider 追加時にこの判定ロジックを触る必要は無い（`launchType` を宣言するだけ）。

`CreatePaymentSessionParams` には `paymentMethod: string` が必ず含まれる。プロバイダはこの値を `getProviderApiMethodId(providerName, methodId)` で API 固有 ID に変換して使用する（後述）。

### PaymentSession と LaunchInstruction

`createSession` の戻り値は `LaunchInstruction` を内包する `PaymentSession` 型。
クライアント側の `executePaymentLaunch` が `instruction.type` で分岐してハンドラに dispatch する。

```typescript
type PaymentSession = {
  sessionId: string;
  instruction: LaunchInstruction;
  expiresAt?: Date;
};

type LaunchInstruction =
  | { type: "redirect"; url: string }
  | { type: "client_sdk"; sdkName: string; config: Record<string, unknown> };
```

- **redirect 型**: provider のホスト型ページ URL（or 自社内ページ URL）を `url` に設定して返す。
- **client_sdk 型**: クライアント側 SDK ローダーレジストリ (`sdkLaunchers`) のキーを `sdkName` に、SDK 起動に必要なパラメータ（publicKey, payload 用情報等）を `config` に詰めて返す。サーバー秘密 (`sk_*`) を `config` に含めてはならない。

### PaymentResult の status フィールド

```typescript
type PaymentResult = {
  success: boolean;
  status?: "completed" | "failed" | "pending";  // Webhook の3分岐に必須
  sessionId: string;
  transactionId?: string;
  paymentMethod?: string;
  paidAt?: Date;
  paidAmount?: number;  // 金額照合用
  rawResponse?: unknown;
  errorCode?: string;
  errorMessage?: string;
};
```

**status による分岐動作:**

- `"completed"` → `completePurchase` を実行
- `"failed"` → `failPurchase` を実行
- `"pending"` → 何もせず 200 を返す（未確定ステータスのスキップ）
- status 未指定時 → `success` で判定（後方互換）

## 追加手順

### 1. プロバイダクラスの作成

`payment/<provider-name>/` ディレクトリを作成し、`PaymentProvider` インターフェースを実装する。

```typescript
// src/features/core/purchaseRequest/services/server/payment/<provider>/<provider>Provider.ts

import type {
  PaymentProvider,
  CreatePaymentSessionParams,
  PaymentSession,
  PaymentResult,
  PaymentStatusResult,
} from "@/features/core/purchaseRequest/types/payment";

export class ExamplePaymentProvider implements PaymentProvider {
  readonly providerName = "example";

  /**
   * 決済セッションを作成
   * sessionId と redirectUrl を返す
   *
   * params.paymentMethod は payment.config.ts の paymentMethods[i].id と一致する値。
   * プロバイダ API 固有 ID への変換は getProviderApiMethodId 経由で行う。
   * mapping が定義されていないメソッドが渡ってきた場合は明示的に throw すること
   * （誤ルートを fail-fast で検知させる）。
   */
  async createSession(params: CreatePaymentSessionParams): Promise<PaymentSession> {
    // 共通 ID → プロバイダ API 固有 ID への変換
    const providerMethodId = getProviderApiMethodId("example", params.paymentMethod);
    const mappingExists =
      paymentConfig.providers.example?.methodMapping?.[params.paymentMethod];
    if (!mappingExists) {
      throw new Error(
        `[Example] 支払い方法 "${params.paymentMethod}" の methodMapping が定義されていません。`,
      );
    }

    // プロバイダAPIで決済セッションを作成（providerMethodId を渡す）
    // ...
    return {
      sessionId: "provider-session-id",
      instruction: {
        type: "redirect",
        url: "https://provider.com/pay/xxx",
      },
      expiresAt: new Date(Date.now() + 30 * 60 * 1000),
    };
  }

  /**
   * Webhookペイロードを検証・パースし PaymentResult を返す
   *
   * 重要: status フィールドで3分岐する
   * - 成功 → status: "completed"
   * - 確定失敗 → status: "failed"
   * - 「成功でも確定失敗でもない」ステータス → 全て status: "pending"
   */
  async verifyWebhook(request: Request): Promise<PaymentResult> {
    const body = await request.text();
    const event = JSON.parse(body);

    // 成功イベント
    if (event.status === "CAPTURED") {
      return {
        success: true,
        status: "completed",
        sessionId: event.sessionId,
        transactionId: event.transactionId,
        paymentMethod: "credit_card",
        paidAt: new Date(event.paidAt),
        paidAmount: event.amount,  // プロバイダが実際に課金した金額
        rawResponse: event,
      };
    }

    // 確定失敗イベント
    if (event.status === "FAILED" || event.status === "CANCELED") {
      return {
        success: false,
        status: "failed",
        sessionId: event.sessionId,
        errorCode: event.errorCode,
        errorMessage: event.errorMessage,
        rawResponse: event,
      };
    }

    // それ以外（AUTHORIZED, PROCESSING 等）→ 何もせずスキップ
    return {
      success: false,
      status: "pending",
      sessionId: event.sessionId ?? "",
      rawResponse: event,
    };
  }

  /**
   * Webhook署名の検証（オプション）
   * 重要: timingSafeEqual を使うこと（タイミング攻撃防止）
   */
  async verifyWebhookSignature(request: Request, signature: string): Promise<boolean> {
    const { timingSafeEqual } = await import("crypto");
    const body = await request.clone().text();
    const expected = computeHmac(body, process.env.EXAMPLE_WEBHOOK_SECRET!);

    return timingSafeEqual(
      Buffer.from(signature),
      Buffer.from(expected)
    );
  }

  /**
   * ポーリング用ステータス照会（オプション）
   */
  async getPaymentStatus(sessionId: string): Promise<PaymentStatusResult> {
    // プロバイダAPIでステータスを照会
    // ...
    return {
      status: "completed",
      sessionId,
      transactionId: "xxx",
      paidAt: new Date(),
    };
  }
}

export const examplePaymentProvider = new ExamplePaymentProvider();
```

### 3. payment.config.ts への設定追加

設定は `providers`（プロバイダ自体の有効/無効と API 名変換）と `paymentMethods`（ユーザーが選べる支払い方法と担当プロバイダ）の **2 箇所** を更新する。

#### 3-1. `providers` への登録

プロバイダの `enabled` フラグと、共通メソッド ID → プロバイダ API 固有 ID への `methodMapping` を定義する。

```typescript
// src/config/app/payment.config.ts

export const paymentConfig = {
  providers: {
    // ... 既存プロバイダ
    example: {
      enabled: true,
      // 共通 paymentMethod ID → プロバイダ API 固有 ID への変換マップ。
      // このプロバイダで担当する全メソッドを列挙する。
      // mapping が無いメソッドは createSession 内で fail-fast させること（誤ルート防止）。
      methodMapping: {
        credit_card: "card",        // Stripe 例: "card"
        convenience_store: "konbini", // Fincode 例: "Konbini" 等、プロバイダ仕様に合わせる
      },
    },
  },

  webhook: {
    signatureHeaders: {
      // ... 既存設定
      example: "X-Example-Signature",  // Webhook署名ヘッダー名
    },
  },

  // ペイロード全体のログ出力制御
  // 本番では false にすること
  debugLog: false,
};
```

#### 3-2. `paymentMethods` への登録（このプロバイダが新メソッドを担当する場合）

ユーザーが UI で選択する支払い方法を `paymentMethods` に登録する。**`provider` フィールドが担当プロバイダを指す**（1メソッド=1プロバイダ）。既存メソッドの担当を切替えるだけならここだけ書き換える。

```typescript
paymentMethods: [
  // ... 既存メソッド
  {
    id: "credit_card",                     // クライアント・サーバー間で使う共通 ID
    label: "クレジットカード",                // UI 表示用
    description: "VISA / Mastercard / JCB", // UI 補足
    icon: "credit-card",                    // SupportedPaymentMethods のアイコン識別子
    status: "available",                    // available / coming_soon / disabled
    provider: "example",                    // ← このメソッドを処理するプロバイダ（status="available" 時必須）
  },
] as PaymentMethodConfig[],
```

**重要な制約**:

- `paymentMethods[i].provider` は `providers` に存在する有効なプロバイダ名であること
- 同一の `id` を複数のメソッドエントリで使用しない（ID は globally unique）
- 同一の `id` を複数プロバイダで担当することは禁止（1メソッド=1プロバイダ）
- `methodMapping` のキーは、当該プロバイダが担当する `paymentMethods[i].id` と一致させること

**注意**: ペイロード全体のログ出力は `paymentConfig.debugLog` で制御する。本番環境では必ず `false` に設定すること。

#### 3-3. プロバイダ解決の仕組み（参考）

ユーザーが選んだ `paymentMethod` から担当プロバイダへの解決は `initiatePurchase` 内で以下の優先順位で行われる:

1. `params.paymentProvider` が明示指定されていればそれを使用（A/B テスト等の特殊経路用）
2. `resolveProviderForMethod(paymentMethod)` で `paymentMethods[i].provider` から解決

`isPaymentMethodSelectable` を通過した paymentMethod に対して 2 は必ず成功する。downstream 側でルーティングを変えたい場合は `paymentMethods[i].provider` を書き換えるだけで配信先が切替わる（コード変更不要）。

### 4. Webhook識別子リゾルバー（必要な場合）

`verifyWebhook` が返す `sessionId` が `createSession` で返した `sessionId` と異なる場合に必要。

例: Fincode は `order_id`（UUID短縮形）を返すため、`provider_order_id` カラムで検索する。

```typescript
// src/features/core/purchaseRequest/services/server/wrappers/purchaseHelpers.ts

// リゾルバーマップに追加
const webhookIdentifierResolvers: Partial<
  Record<PaymentProviderName, WebhookIdentifierResolver>
> = {
  fincode: resolveFincodeIdentifier,
  example: resolveExampleIdentifier,  // ← 追加
};

// リゾルバー関数
async function resolveExampleIdentifier(
  identifier: string
): Promise<PurchaseRequest | null> {
  // provider_order_id で検索する例
  const results = await purchaseRequestDrizzleService.search({
    where: { provider_order_id: identifier },
  });
  return results.data[0] ?? null;
}
```

`provider_order_id` カラムを活用する場合は、provider 契約の **`deriveProviderOrderId(purchaseRequestId)`**
を実装すること（`initiatePurchase` が `buildLaunchForRequest` 内で自動的に保存する。provider 名の
直書き分岐は不要）。あわせて **`correlationKey: "order_id"`** を宣言すると、ステータス照会
（`getPurchaseStatusForUser`）も `provider_order_id` を照合キーに使う。

### 5. 環境変数の設定

プロバイダの API キーや Webhook シークレット等を `.env.development` / `.env.production` に追加する。
プロバイダの選択は `payment.config.ts` の `paymentMethods[i].provider` で行うため、選択用の環境変数は不要。

```env
# .env.development / .env.production

# プロバイダ固有の設定
EXAMPLE_API_KEY=xxx
EXAMPLE_SECRET_KEY=xxx
EXAMPLE_WEBHOOK_SECRET=xxx
```

### 6. Webhookエンドポイント

既存のエンドポイントをそのまま使用する。新規作成は不要。

```
POST /api/webhook/payment?provider=<name>
```

- `webhookHandler.ts` の `handleWebhook` が provider クエリパラメータに基づいて自動で振り分け
- プロバイダのダッシュボードで以下のURLを設定:
  - 開発: `https://<ngrok-url>/api/webhook/payment?provider=example`
  - 本番: `https://your-domain.com/api/webhook/payment?provider=example`

**エラーハンドリング:**

- `DomainError` → 200（プロバイダ側のリトライを防止）
- 500系エラー → 500（プロバイダ側がリトライ）

### 7. 本番環境での注意事項

- `dummy` プロバイダは `NODE_ENV=production` でブロックされる
- Host Header Injection 防止: `businessConfig.url` を使用（リクエストヘッダーからホスト名を取得しない）
- PII ログ: `paymentConfig.debugLog = false` を確認

## client_sdk 型の追加差分

paidy.js のようにブラウザ JS SDK を読み込んでモーダル決済を起動するタイプのプロバイダは、redirect 型と動線が異なるため追加ファイルが必要。**redirect 型と共通の部分**（プロバイダクラス・ファクトリ登録・payment.config.ts 設定・環境変数）は上記手順をそのまま踏襲し、その上で以下を追加する。

### A. createSession の戻り値

外部 provider の URL を返さず、自社内で組み立てた起動指示を返す。Paidy 実装例:

```typescript
return {
  sessionId: params.purchaseRequestId,  // Paidy 側にまだ payment が存在しないため流用
  instruction: {
    type: "client_sdk",
    sdkName: "paidy",                   // クライアント sdkLaunchers レジストリのキー
    config: {
      publicKey: config.publicKey,      // pk_*（クライアントに渡される）
      purchaseRequestId: params.purchaseRequestId,
      amount: params.amount,
      currency: "JPY",
      // 与信通過率を上げるための補助情報（ユーザー名・メアド・電話番号等）
      buyerName: user?.name ?? "購入者",
      buyerEmail: params.buyerEmail,
      buyerPhoneNumber: params.buyerPhoneNumber,
    },
  },
  expiresAt: new Date(Date.now() + 30 * 60 * 1000),
};
```

`config` はクライアントに直接渡るため、サーバー秘密（`sk_*`、Webhook secret 等）を含めてはならない。

### D. 確定サービスを provider ディレクトリに追加

`payment/<provider>/confirm<Name>Payment.ts` を作成。確定 API ルートから呼ばれる本体ロジック:

1. `purchase_request` 取得 + 認可（user_id + provider 整合性）
2. ステータス分岐（completed なら冪等 return、processing 以外は 400）
3. provider 側 API への二重確認（amount / status 検証）
4. 必要に応じて capture を能動的に実行
5. `payment_session_id` に provider 側 payment_id を保存（Webhook 照合用）
6. `completePurchase` 呼び出し（楽観ロック 409 は alreadyCompleted で吸収）
7. `auditLogger.record({ action: "purchase_request.payment.confirmed", ... })`

詳細は `payment/paidy/confirmPaidyPayment.ts` を参照。

### E. 動線まとめ

```
[client] 購入ボタン
   ↓ initiate API
[server] purchase_request 作成 + LaunchInstruction.client_sdk 返却
   ↓
[client] executePaymentLaunch → clientSdkHandler
   ↓ sdkLaunchers[sdkName].load() + launch(config)
[client] SDK モーダル起動 → ユーザー決済 → authorized
   ↓ keepalive fetch（fire & 即時遷移）
[server] /api/wallet/purchase/[id]/<provider>/confirm → confirm<Name>Payment
   │   ↓ provider API 二重確認 + capture + completePurchase
   ↓
[client] callback 画面（successUrl）→ ポーリング → 完了画面
   ↓ 並行で Paidy 側から
[server] /api/webhook/payment?provider=<name> → verifyWebhook（フォールバック）
```

### F. client_sdk 型固有のチェックリスト

redirect 型のチェックリストに加えて:

- [ ] `createSession` が `{ type: "client_sdk", sdkName, config }` を返す
- [ ] `config` にサーバー秘密（sk_*、Webhook secret 等）が含まれていない
- [ ] `sdkLaunchers/<name>.ts` を作成して `SdkLauncher` を実装
- [ ] `sdkLaunchers/index.ts` のレジストリにエントリを追加
- [ ] 確定 API ルート `app/api/wallet/purchase/[id]/<name>/confirm/route.ts` を作成
- [ ] `payment/<name>/confirm<Name>Payment.ts` を作成（認可 + provider API 二重確認 + capture + completePurchase）
- [ ] provider API が大文字 / paidy.js コールバックが小文字のように status 値の表記が混在する場合は正規化ヘルパーを用意

## トラブルシューティング

### Webhook が届かない

- ngrok が起動しているか確認
- プロバイダのダッシュボードで URL が正しいか確認
- 必要なイベントが有効になっているか確認

### 署名検証エラー（401）

- 環境変数の署名シークレットが正しいか確認
- `payment.config.ts` の `signatureHeaders` が正しいか確認

### セッションが見つからない

- `payment_session_id` が DB に保存されているか確認
- リゾルバーが正しく実装されているか確認

### PENDING で失敗扱いになる

- `verifyWebhook` で `status: "pending"` を返しているか確認
- 「成功でも確定失敗でもない」ステータスは全て `"pending"` にする

### 金額不一致エラー

- `paidAmount` と `payment_amount` が一致するか確認
- プロバイダ側で実際に課金された金額を正しく取得しているか確認
