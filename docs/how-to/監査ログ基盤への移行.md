# 監査ログ基盤への移行ガイド

旧 `user_action_logs` テーブル / `userActionLog` ドメインを廃止し、新しい
汎用 `audit_logs` 基盤へ移行する手順。

> **対象**: テンプレートをフォークして稼働中のダウンストリームプロジェクト。
> 新規セットアップ (`audit_logs` を最初から使うプロジェクト) では不要。

## 全体像

```
[1] git pull            → 新 audit_logs テーブル定義と移行 script を取り込む
[2] pnpm db:push        → audit_logs / audit_logs_failed テーブル作成
[3] audit:migrate       → 過去ログを user_action_logs から audit_logs へコピー
[4] 動作確認            → /admin/audit-logs と各ユーザーの履歴タブをスポット確認
[5] audit:drop-legacy   → user_action_logs テーブルを物理削除
[6] git pull            → userActionLog ドメインのコードがテンプレートから削除済み
```

ステップ [3] と [5] を分離している理由: 検証前に旧テーブルを削除する事故を
構造的に防ぐため。

## 前提

- `auditLogger.record()` 経由の新規ログは既に `audit_logs` に書き込まれる状態
  (テンプレートを取り込み済み)。
- `user_action_logs` には移行が必要な過去ログが残っている。

## 手順

### 1. 新テーブル作成

```bash
pnpm db:push
```

`audit_logs` / `audit_logs_failed` と enum `audit_actor_type` が作成される。
既に作成済みなら no-op。

### 2. dry-run で移行内容を確認

```bash
pnpm audit:migrate -- --dry-run
```

- 何件が変換対象か
- 未知の `action_type` が混ざっていないか
- 既に移行済みのレコードがあれば skip 件数

を **DB に書き込まずに** 確認できる。

### 3. 本番実行

```bash
pnpm audit:migrate
```

- バッチサイズ 1000 件単位で進捗ログを出力
- 既に audit_logs に同じ `(target_id, action, created_at)` のレコードが
  あれば skip → **再実行可能 (冪等)**
- 大規模データの場合は `--limit 100` で部分的に試してから全件実行を推奨

`context` 列には `{ migrated: true, sourceTable: "user_action_logs", sourceId: <旧 ID> }`
が入るので、移行されたレコードは後から特定可能。

### 4. 動作確認

ダッシュボードから以下をスポット確認:

- `/admin/audit-logs` 横断検索画面
  - 件数が `pnpm db:count user_action_logs` と一致するか (target_type=user で絞り込み)
  - 過去 action のラベルが表示されているか
- 任意のユーザー管理モーダルの「履歴」タブ
  - 過去のステータス変更/退会等が表示されるか
  - 行クリック → 詳細モーダルで before/after が表示されるか

問題があれば `audit_logs` テーブルから DELETE 可能 (旧 `user_action_logs`
は無傷で残っているため再実行できる)。

### 5. 旧テーブル削除

```bash
# まず影響件数を dry-run で確認
pnpm audit:drop-legacy -- --dry-run

# 問題なければ実行 (--confirm 必須)
pnpm audit:drop-legacy -- --confirm
```

安全策:

- `audit_logs` に user 系レコードが 1 件もない場合は停止 (移行未実施の疑い)
- `--confirm` がなければ停止

実行内容: `DROP TABLE user_action_logs CASCADE` + 関連 enum (`user_action_type`,
`user_action_actor_type`) の DROP。

### 6. 旧ドメインのコード削除

テンプレートの後続リリースで `src/features/core/userActionLog/` ディレクトリと
registry 登録 (`schemaRegistry.ts` / `serviceRegistry.ts` / `tsconfig.json` の
paths) が削除される。`git pull` で取り込めば完了。

ダウンストリーム側で旧ドメインに直接 import しているコードがあれば、
事前に `auditLogger` 経由 / `AuditTimeline` 経由に書き換えておくこと。

## トラブルシュート

### 未知の action_type が混ざっている

旧 `user_action_logs.action_type` enum をフォーク先で拡張している場合、
script のマッピング表に無い値が混ざる。デフォルトでは停止するので、以下のどちらか:

**方針 A**: マッピングを追加する
`scripts/audit-logs/migrate-from-user-action-logs.ts` の `ACTION_TYPE_MAP` に
追記する。新 action 名は `<domain>.<entity>.<verb_past>` 規約に従う。

**方針 B**: フォールバックに任せる

```bash
pnpm audit:migrate -- --allow-unknown
```

未知の値は `user.legacy_<旧名>` という action として記録される。表示時の
ラベルは元の文字列がそのまま出る (規約違反だが動作はする)。

### `audit_logs` 件数が `user_action_logs` より多い

新規ログが移行と並行して書き込まれた可能性。スポット確認で過去 created_at
範囲のレコードが正しいかどうかが本質的な検証ポイント。

### 移行後に巻き戻したい

`audit_logs` から移行レコードのみ削除:

```sql
DELETE FROM audit_logs WHERE context->>'migrated' = 'true';
```

ただし旧 `user_action_logs` を既に DROP していたら、過去ログは復元不能。
**旧テーブル DROP 前に必ず動作確認すること。**
