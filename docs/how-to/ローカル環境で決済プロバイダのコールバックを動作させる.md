# ローカル環境で決済プロバイダのコールバックを動作させる

ローカル開発環境（`localhost:3000` など）で Fincode / Square 等のリダイレクト型決済プロバイダのコールバックを正しく着地させるための設定手順。

## 背景

決済セッション作成時、サーバから決済プロバイダに渡す `success_url` / `cancel_url` は、**ユーザーが現在ブラウザで開いているドメイン**ではなく、**サーバ側で確定した「アプリのベースURL」** から組み立てる必要がある。

これは Host Header Injection 対策（リクエストヘッダ由来の URL を信用すると、攻撃者がヘッダを偽装してコールバック URL を任意のドメインに向けられる）のため。実装としては `getAppBaseUrl()` (`src/lib/url/getAppBaseUrl.ts`) が単一の入口になっている。

`getAppBaseUrl()` の優先順位:

1. `process.env.APP_BASE_URL` — 明示的上書き（最優先）
2. Vercel preview デプロイの自動検出 (`VERCEL_URL`)
3. `businessConfig.url` — 本番環境のフォールバック

ローカル環境では 1 を使う。

## 手順

### 1. `.env.development` に `APP_BASE_URL` を設定

```
APP_BASE_URL=http://localhost:3000
```

ポート番号はプロジェクトの `PORT` と一致させること。これでローカルから決済を開始した場合、`success_url` がローカルホストを指すようになり、決済プロバイダ側からのリダイレクトが正しく着地する。

### 2. Webhook を最後まで通したい場合は ngrok 等のトンネルを使う

上記までで「ユーザーがコールバックページに着地する」ところまでは動くが、**決済プロバイダからの Webhook はインターネット経由で到達するため `localhost` には届かない**。Webhook → ステータス更新 → 完了画面までを通したい場合は、ローカルホストを公開するトンネルが必要。

#### ngrok 使用例

```bash
ngrok http 3000
```

ngrok が割り当てた URL（例: `https://xxxxxxxx.ngrok-free.app`）を `.env.development` に反映:

```
APP_BASE_URL=https://xxxxxxxx.ngrok-free.app
```

その上で、各決済プロバイダのテストモードダッシュボードにも同じドメインの Webhook URL を登録する:

- **Fincode テストモード** (`dashboard.test.fincode.jp`)
  - Webhook 設定 → `https://xxxxxxxx.ngrok-free.app/api/webhook/payment?provider=fincode`
- **Square Sandbox** (`developer.squareup.com`)
  - Webhook 設定 → `https://xxxxxxxx.ngrok-free.app/api/webhook/payment?provider=square`

ngrok の URL は再起動のたびに変わる（無料プランの場合）ので、その都度 `.env.development` とプロバイダダッシュボードの両方を更新する必要がある。

### 3. 簡易確認: dummy プロバイダを使う

実プロバイダ通信を伴わない UI フロー確認だけなら `dummy` プロバイダで十分:

```
PAYMENT_PROVIDER=dummy
```

`dummy` プロバイダはサーバ内で完結するため Webhook 設定は不要。ただし `NODE_ENV=production` ではブロックされる（コミット 3b18deeb）ので本番では使用不可。

## 推奨運用

- **通常の開発**: `PAYMENT_PROVIDER=dummy` で UI フローと完了画面までを高速に確認
- **プロバイダ結合確認**: ngrok + `APP_BASE_URL` + プロバイダ側 Webhook 登録の3点セット
- **本番同等の実決済テスト**: staging 環境（カスタムドメイン付き）で実施

dev で実プロバイダの結合まで通そうとすると ngrok URL とダッシュボード設定の同期が必要で、開発体験が悪い。staging 環境を持っているならそちらで実プロバイダ結合確認するのが現実的。
