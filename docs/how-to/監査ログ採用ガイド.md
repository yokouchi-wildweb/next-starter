# 監査ログ採用ガイド

汎用監査ログ基盤 (`src/lib/audit` + `src/features/core/auditLog`) を各ドメインで利用するための手順書。

upstream で導入済みの基盤に対し、フォーク先プロジェクトでドメインを段階的にオプトインしていく前提で書かれている。テンプレート同梱のため新規プロジェクトはそのまま使える。

## 何を記録するか

`audit_logs` テーブルは「**誰が** / **いつ** / **どのレコードに** / **誰に対して** / **どんな操作をしたか** / **何が変わったか**」を append-only で残す。

- target_type / target_id: 監査対象の domain と record（例: `"user"` / `<userId>`、`"wallet"` / `<walletId>`）
- **subject_user_id**: 「この操作が誰のユーザーに対するものか」（"data subject"）の集約キー
- action: `"<domain>.<entity>.<verb>"` の規約（例: `"user.email.changed"`）
- before_value / after_value: 変更フィールドのみ（trackedFields でフィルタ済み）
- actor_id / actor_type: 実行者（admin / user / system / api_key / webhook）
- context: ip / user_agent / session_id / request_id（`createApiRoute` で自動敷設）
- retention_days: 行単位の保持期間（cron で経過行を自動削除）

書き込み失敗時に行が消えないよう、`bestEffort: true` で記録した場合は `audit_logs_failed` (dead-letter) に退避され、cron で再投入される。

## actor vs subject（実行者と対象ユーザーの概念分離）

監査ログには「**actor (誰が操作したか)**」と「**subject (誰に対する操作か)**」という **2 つの異なる軸**がある。これを分離して記録することで、後段で「このユーザーへの全操作」「この管理者が行った全操作」のどちらの集約も index 駆動で取れる。

| 軸 | カラム | 意味 |
|---|---|---|
| actor | `actor_id` / `actor_type` | 操作を実行した主体（管理者 / システム / ユーザー本人 / webhook 等） |
| subject | `subject_user_id` | 操作対象のユーザー（"data subject"、GDPR 用語） |
| target | `target_type` / `target_id` | 操作対象の **レコード**（user / wallet / user_item 等） |

### subject_user_id の設定ルール

| target_type の例 | subject_user_id に設定する値 |
|---|---|
| `"user"` | `target_id` と同じユーザー ID（冗長記録） |
| `"wallet"` | そのウォレットが属する `userId` |
| `"user_item"` | そのユーザーアイテムが属する `userId` |
| `"subscription"` | そのサブスクリプションの契約 `userId` |
| `"bank_transfer_review"` | 振込元の `userId`（特定可能な場合） |
| bulk aggregate（複数ユーザー対象） | `null`（対象ユーザーが単一でないため） |
| システム設定変更（全ユーザー影響） | `null` |

**規約**: 「ユーザーに紐づく操作」では `subjectUserId` を**必ず**設定する。`metadata.userId` に隠すのは禁止（後段で jsonb スキャンになる + 入れ忘れが構造的に検知できない）。`auditLogger.record({ subjectUserId })` をトップレベルフィールドとして明示する。

### なぜこの分離が必要か

- **`target_type='user', target_id=$userId` だけでは関連エンティティ操作が拾えない**: 管理者がそのユーザーのウォレットを調整した場合、レコードは `target_type='wallet', target_id=<walletId>` で記録される。「このユーザーへの全操作」を出すには target だけでは不十分。
- **`metadata.userId` 方式の構造的脆弱性**: jsonb の中に隠すと、index を効かせるのが面倒（GIN expression index になる）、かつ「設定し忘れた」場合に検出不能。トップレベルカラムに昇格させることで、ESLint / 型 / partial index の 3 段で守る。
- **GDPR 等の法令対応**: 「あるユーザーに関連する全データ」の抽出要求（削除権・アクセス権）が来た際、subject_user_id で 1 クエリで全 audit を取得できる必要がある。

### bulk aggregate との関係

`bulkAdjustByType` / `bulkAdjustByUsers` のような bulk 操作で `bulkMode: aggregate` を採用している場合、**1 行に 1 ユーザーは対応付かない**ため `subject_user_id` は **null** になる。これによりユーザー詳細画面のアクティビティタイムラインには表示されない。

bulk 介入もユーザー単位のタイムラインに出したい場合は **detail モード（`recordMany` で N 行記録）** に切り替える設計判断が必要。コストとのトレードオフで、ドメインごとに判断する。

### クエリパターン

```sql
-- このユーザーに対する全操作（target_type を跨ぐ）。partial index で高速
SELECT * FROM audit_logs
WHERE subject_user_id = $1
ORDER BY created_at DESC;

-- この管理者が行った全操作
SELECT * FROM audit_logs
WHERE actor_id = $1
ORDER BY created_at DESC;

-- 特定の record の履歴
SELECT * FROM audit_logs
WHERE target_type = $1 AND target_id = $2
ORDER BY created_at DESC;
```

UI コンポーネントとしては:

- `<AuditTimeline targetType targetId />`: 特定の record の履歴（target 軸）
- `<UserActivityTimeline userId />`: ユーザー単位の集約タイムライン（subject 軸）

の 2 系統を用途で使い分ける。

## audit_logs に載せる / 載せないの判断軸（「介入 vs 業務」）

## audit_logs に載せる / 載せないの判断軸（「介入 vs 業務」）

audit_logs は「**介入 (intervention)**」操作のみを記録する。「**業務 (business)**」操作は載せない。これを混同すると、audit テーブルが業務イベントで埋もれて統制目的が果たせなくなる + ストレージ・I/O コストが膨らむ。

### 「介入 (intervention)」として audit に載せる

- 管理者が業務フロー外で手動操作する（actor_type=admin、admin 専用 API 経由）
- ポリシーを曲げる・緊急対応・手動修正
- ユーザーの財産や権限に直接介入する操作（残高調整、ロール変更、強制状態遷移など）

### 「業務 (business)」として audit に載せない

- システム設計通りに自動実行される高頻度処理（ガチャ消費、購入決済確定、通知送信など）
- ユーザー操作の正常系（actor_type=user による課金・消費・コンテンツ閲覧）
- 既に業務テーブル（`wallet_histories`、`order_histories`、ドメイン固有の履歴テーブル等）で追跡可能なもの

### 迷ったときの判断手順

1. **その関数が `actor_type=admin` から呼ばれる前提が 1 つでもあるか?**
   - Yes → 「介入」→ audit に載せる
   - No（system / user のみ） → 「業務」→ 載せない
2. **同じ関数が admin / system 両方から呼ばれる場合は載せる**
   - actor_type で後から `WHERE actor_type='admin'` 等で絞れるため、admin パスがあれば常に記録
3. **件数規模が大きい bulk 介入は `aggregate` モード（1 操作 1 行）で集約記録する**
   - 詳細は `bulkMode` セクションを参照
4. **三重記録を避ける**
   - 業務テーブル + ドメイン履歴テーブル + audit_logs に同じ事実を冗長記録するくらいなら、業務テーブルで完結させる

### 具体例: ウォレット操作

| 操作 | actor | 載せる/載せない | 理由 |
|------|-------|----------------|------|
| 管理者による残高調整 (`adjustBalance`) | admin | ✅ 載せる | 管理者の介入操作 |
| ソフトデリート時の残高クリア (`clearBalance`) | admin/system | ✅ 載せる | 業務フロー外のクリーンアップ |
| ガチャ消費 (`consumeReservedBalance`) | user/system | ❌ 載せない | 業務フローで `wallet_histories` 記録済み |
| 即時引落 (`debitBalance`) | system | ❌ 載せない | 同上 |
| 残高予約 / 解除 (`reserveBalance` / `releaseReservation`) | system | ❌ 載せない | 残高変動なし（locked_balance のみ） |

詳しくは [`src/features/core/wallet/README.md`](../../src/features/core/wallet/README.md) の「監査ログポリシー」セクション参照。

### 横展開: 他ドメインでも同じ原則を適用する

同じ判断軸は `userItem` / `couponGrant` / `subscription` / `point` 系など、他のドメインでも適用する。たとえば:

- **`userItem` 一括ステータス変更**: 管理者から admin API 経由 → 「介入」→ aggregate 記録
- **`userItem` 通常配布フロー**: ユーザー購入や報酬受け取り → 「業務」→ ドメイン履歴で代替
- **`subscription` 手動キャンセル**: 管理者がユーザー代理でキャンセル → 「介入」→ 記録
- **`subscription` 自動更新**: cron による定期処理 → 「業務」→ 載せない（更新履歴テーブルで追跡）

## 採用パターン

### パターン A: createCrudService の `audit:` で CRUD を自動記録

最小コストで CRUD ライフサイクル全てを拾いたい場合。`drizzleBase.ts` で設定するだけ。

```ts
// src/features/<domain>/services/server/drizzleBase.ts

import { auditLogger } from "@/features/core/auditLog";
import { createCrudService } from "@/lib/crud/drizzle";
// ...

export const baseOptions = {
  // ... 既存オプション
  audit: {
    enabled: true,
    targetType: "post",
    actionPrefix: "post",
    trackedFields: ["title", "body", "status"],
    bulkMode: "aggregate", // 既定。重要データは "detail" を選ぶ
    retentionDays: 365,    // 省略可。デフォルト 365 日
    strict: false,         // 既定 true。false にすると失敗時 dead-letter 退避
    recorder: auditLogger,
  },
} satisfies DrizzleCrudServiceOptions<...>;
```

これだけで以下が自動記録される:

| 操作 | action | mode |
|---|---|---|
| `base.create()` | `post.created` | 単件 |
| `base.update()` | `post.updated` | 差分（trackedFields に絞られる） |
| `base.remove()` | `post.deleted` | 単件 |
| `base.upsert()` | `post.upserted` | 単件 |
| `base.bulkDeleteByIds()` 等 | `post.bulk_deleted` | aggregate / detail / off |
| `base.hardDelete()` | `post.hard_deleted` | 単件 |
| `base.restore()` | `post.restored` | 単件 |

### パターン B: wrapper で `auditLogger.record()` 直叩き

ライフサイクルの一部だけ記録したい / カスタム action 名を付けたい場合。

```ts
// src/features/<domain>/services/server/wrappers/changeStatus.ts

import { auditLogger } from "@/features/core/auditLog";

export async function changeStatus(id: string, nextStatus: string, reason?: string) {
  return await db.transaction(async (tx) => {
    const before = await getById(id, tx);
    const after = await update(id, { status: nextStatus }, tx);

    await auditLogger.record({
      targetType: "post",
      targetId: id,
      subjectUserId: after.author_id,  // ユーザーに紐づく操作なら必ず設定
      action: "post.status.changed",
      before: { status: before.status },
      after: { status: after.status },
      reason,
      tx,             // 必ず同一 tx に乗せる（rollback 連動のため）
    });

    return after;
  });
}
```

両パターンは併用可能。**A で広く拾い、B でドメイン固有のライフサイクルイベントを付ける** のが推奨形。

### パターン C: バッチ記録（`recordMany` / `recordManyDiff`）

同一トランザクション内で同一 action を多数（数十件〜数千件）記録するケースでは、`auditLogger.record()` を逐次 `await` するとラウンドトリップが累積する。Neon HTTP ドライバーでは 1 INSERT あたり数十〜数百 ms かかるため、件数次第で axios のデフォルトタイムアウト（30 秒）に到達してしまう。

このパターン（例: バルク状態遷移、N 件の `UserItem` 同時更新）では `recordMany` / `recordManyDiff` を使うと **1 ステートメント INSERT** に集約される（巨大バッチは内部で自動チャンキング）。

```ts
// 旧（タイムアウトの原因）
for (const row of changed) {
  await auditLogger.record({
    targetType: "user_item",
    targetId: row.id,
    action: "user_item.stock_status.changed",
    before: { stock_status: row.before },
    after: { stock_status: row.after },
    reason,
    tx,
  });
}

// 新
await auditLogger.recordMany(
  changed.map((row) => ({
    targetType: "user_item",
    targetId: row.id,
    action: "user_item.stock_status.changed",
    before: { stock_status: row.before },
    after: { stock_status: row.after },
    reason,
    tx,
  })),
);
```

差分計算を内部で行いたい場合は `recordManyDiff`:

```ts
await auditLogger.recordManyDiff(
  changed.map((row) => ({
    targetType: "user_item",
    targetId: row.id,
    action: "user_item.fields.changed",
    before: row.beforeSnapshot,
    after: row.afterSnapshot,
    trackedFields: ["stock_status", "price"],
    reason,
    tx,
  })),
);
// 差分なしの行は自動 skip。全件 skip された場合 INSERT は発行されない。
```

#### 制約

- 入力配列内で **`tx` が混在しているとエラー**（ロールバック境界が壊れるため throw）
- 入力配列内で **`bestEffort` が混在しているとエラー**（バッチ単位で同じモードを期待）
- 入力配列内で **異なる `batchId` が明示混在しているとエラー**（共通バッチを期待）
- 空配列 / 全件 skip の場合は即 return（INSERT 発行なし）

#### チャンキングと atomic 性

PostgreSQL の prepared statement バインドパラメータ上限（65535）を回避するため、内部で 1000 行/INSERT に自動分割される。チャンク全体の atomic 性は以下のルールで担保される:

| caller の `tx` | チャンク数 | 失敗時の挙動 |
|---|---|---|
| あり | 1 以上 | caller の tx が abort → caller の commit が失敗して全 rollback |
| なし | 1 | 単一 INSERT のため通常通り |
| なし | 2 以上 | `recordMany` 内部で `db.transaction()` を開始し、全 chunk を 1 tx に集約 |

つまり「先頭 chunk は commit 済み、末尾 chunk が失敗」によるスプリット状態は発生しない。

#### トレース性（`batch_id` 列）

`recordMany` 呼び出しごとに UUID v4 の `batchId` が共通発番され、各行の `batch_id` 列（uuid, nullable）に永続化される。部分インデックス `audit_logs_batch_id_idx` により以下のクエリが index scan で動作する:

```sql
-- 特定バッチの全行を抽出（高速）
SELECT * FROM audit_logs
WHERE batch_id = '<uuid>'
ORDER BY created_at;
```

論理的に親バッチに紐づけたい単件記録は `record` / `recordDiff` の `batchId` を明示することで同じグループに含められる:

```ts
const batchId = crypto.randomUUID();

await auditLogger.recordMany(rows.map((r) => ({ ..., batchId })));

// 関連する単件記録も同じ batchId に紐付け
await auditLogger.record({
  targetType: "user_item",
  targetId: parentId,
  action: "user_item.bulk_operation.completed",
  metadata: { affectedCount: rows.length },
  batchId,
  tx,
});
```

`recordMany` に明示的な `batchId` を渡さなければ自動発番される（推奨パターン）。

#### 失敗時の挙動

| 段階 | strict（既定） | bestEffort |
|---|---|---|
| `tx` / `bestEffort` / `batchId` 不整合 | throw | throw（握り潰さない） |
| ALS context 解決失敗 | throw | 入力配列全体を 1 行で dead-letter 退避 |
| 個別 input の buildPayload 失敗 | throw | 該当 input のみ dead-letter、残りは続行 |
| バッチ INSERT 失敗 | throw → tx rollback | 構築済み全 payload を **batched INSERT で** 一括 dead-letter 退避 |

INSERT 失敗時の dead-letter 退避自体もバッチ INSERT に集約しているため、N 回逐次 await による二次タイムアウトは発生しない。dead-letter された行は `batch_id` を保持しており、`recoverDeadLetter` cron で再投入後も同バッチを再結合できる。

## action 命名規約（**ESLint で強制**）

正規表現: `^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)+$`

- 小文字 + 数字 + `_` のみ、ドット区切り
- 最低 2 セグメント（`<domain>.<verb>` または `<domain>.<entity>.<verb>`）

良い例:

- `user.registered`
- `user.email.changed`
- `post.status.changed`
- `wallet.coin.spent`

悪い例:

- `User.Created`（大文字）
- `user-registered`（ハイフン）
- `userRegistered`（区切りなし）
- `user.created.by_admin.with.reason`（過剰なネスト）

`createCrudService` の `actionPrefix` も同じ規約に従う（`.created` 等が後ろに付く想定）。

ESLint で静的に検出される（runtime でも `validateActionName` が throw）。動的に組み立てる必要がある正当なケースは個別 `eslint-disable` で許可する。

## trackedFields の選び方

trackedFields を絞れば、計算と保存量が減る。

- 機微情報・大きな blob を含むカラムは入れない（denylist でも後段除外されるが二重対策推奨）
- 監査として残す価値があるフィールドのみ列挙
- 省略時は **denylist 適用後の全フィールド**

```ts
trackedFields: ["title", "status", "published_at"], // ← ホワイトリスト
```

## bulkMode の選び方

| mode | 1 操作あたりの記録 | 用途 |
|---|---|---|
| `aggregate`（既定） | 1 行（件数 + サンプル ID + 条件スナップショット） | 一般的なドメイン |
| `detail` | N 行（各レコード個別） | 重要データ（user / wallet / payment） |
| `off` | 0 行 | analytics 等の高頻度大量データ |

`detail` は bulk_update で個別 SELECT が必要となるため、I/O コストが大きい。基本は `aggregate` で運用し、必要なドメインだけ `detail` にする。

## retention_days の決め方

| ドメインの性質 | 推奨値 |
|---|---|
| コンプライアンス対象（user / payment / wallet） | 730 日（2年）以上 |
| 一般的な業務データ | 365 日（既定値） |
| 高頻度ログ性データ | 90 日 |

行ごとに保持期間を変えられる（recorder 呼び出し時に `retentionDays` で上書き可能）。

## context の渡し方

`createApiRoute` および `createCronRoute` 由来のリクエストでは `runWithAuditContext` が **自動敷設** される。handler 内の任意の深さから `auditLogger.record()` を呼べば actor / IP / UA / requestId が自動的に紐づく。

API ルート / cron 以外の経路（バックグラウンドジョブ・スクリプト等）では明示的に context を敷設する:

```ts
import { runAsSystem } from "@/lib/audit";

await runAsSystem(async () => {
  // この中で呼ばれた auditLogger.record() は actor_type="system" として記録される
  await someBackgroundTask();
});
```

または完全カスタム:

```ts
import { runWithAuditContext } from "@/lib/audit";

await runWithAuditContext(
  { actorId: "batch-runner", actorType: "system", ip: null, userAgent: null, sessionId: null, requestId: crypto.randomUUID() },
  async () => { /* ... */ },
);
```

## denylist でフィールドを除外する

機密フィールド（パスワード・トークン等）は `src/lib/audit/denylist.ts` に追加する。`stripDenylisted` が before/after から除外する。

denylist は **全ドメイン横断**。ドメイン固有の除外をしたい場合は trackedFields のホワイトリスト方式で代替する。

## AuditTimeline / UserActivityTimeline UI を組み込む

特定の record の履歴を画面に出したい場合（target 軸）:

```tsx
import { AuditTimeline } from "@/features/core/auditLog";

<AuditTimeline targetType="post" targetId={postId} />
```

特定のユーザーに対する全操作を出したい場合（subject 軸、target_type を跨いで集約）:

```tsx
import { UserActivityTimeline } from "@/features/core/auditLog";

<UserActivityTimeline userId={userId} />
```

`UserActivityTimeline` は `audit_logs.subject_user_id` で絞り込むため、`auditLogger.record({ subjectUserId })` が**正しく設定されているレコードのみ**表示される。設定漏れの操作は表示されないので、ユーザー詳細画面の「操作履歴」タブとして組み込む場合は事前に既存 wrapper の subjectUserId 設定状況を確認すること。

横断検索は `/admin/audit-logs` ですでに提供済み。

## action ラベルを日本語表示する

未登録の action 名はそのまま表示される。表示名を整えたい場合:


## 失敗時の挙動

| モード | 失敗時 |
|---|---|
| `strict: true`（既定） | throw → 呼び出し元 tx ごと rollback |
| `bestEffort: true` | dead-letter (`audit_logs_failed`) に退避し、本処理は継続 |

ユーザー操作起点の重要記録は `strict`、analytics 的な記録は `bestEffort` を使う。

## 運用タスク（cron）

DB push 後に以下の cron を有効化する:

| タスク | 推奨スケジュール | 役割 |
|---|---|---|
| `audit-log-prune` | `0 3 * * *`（1 日 1 回） | retention_days を超えた行を削除 |
| `audit-log-recover-dead-letter` | `0 * * * *`（1 時間 1 回） | dead-letter の再投入を試行 |

API ルート: `/api/cron/audit-log-prune`、`/api/cron/audit-log-recover-dead-letter`
CLI: `pnpm cron audit-log-prune`、`pnpm cron audit-log-recover-dead-letter`

dead-letter の件数が時間とともに減らない場合は schema 不整合や validation 違反の可能性があるため、`audit_logs_failed.error_message` を確認する。

## ダウンストリーム移行手順（推奨順）

1. **DB push**: `audit_logs` / `audit_logs_failed` / enum / index を作成
2. **cron 有効化**: 上記 2 タスクをスケジュールに登録
3. **採用方針決定**: パターン A / B / 併用、retention_days、bulkMode をドメインごとに決める
4. **段階移行**: ドメイン単位で `audit:` を有効化、または wrapper を `auditLogger.record()` ベースに書き換え
5. **action ラベル登録**: `registerActionLabels` で日本語化
6. **AuditTimeline 組み込み**: 必要な管理画面で履歴 UI を追加

破壊的変更ではないため、段階的にオプトインしていけばよい。

### upstream 側で schema が更新されたときの追従

upstream で `audit_logs` の列やインデックスが追加された場合、ダウンストリームは **`pnpm db:push` を実行する必要がある**。代表的な追加履歴:

| 追加対象 | 内容 | 移行リスク |
|---|---|---|
| `batch_id` 列 (uuid, nullable) | `recordMany` のバッチ識別子 | 低（nullable 追加・既存行は NULL）|
| `audit_logs_batch_id_idx` (partial) | `WHERE batch_id IS NOT NULL` 部分インデックス | 低（既存クエリへの影響なし）|
| `subject_user_id` 列 (text, nullable) | "data subject"（操作対象ユーザー）の集約キー | 低（nullable 追加・既存行は NULL）|
| `audit_logs_subject_user_id_idx` (partial) | `WHERE subject_user_id IS NOT NULL` 部分インデックス | **中** — 既存テーブルが大きい場合は `CREATE INDEX CONCURRENTLY` 推奨 |

`record` / `recordDiff` の既存呼び出しに変更は不要（`subjectUserId` は optional）。新規 schema を反映しないまま `auditLogger.record({ subjectUserId })` を呼ぶと INSERT が「列 `subject_user_id` が存在しない」エラーで失敗するため、リリース前に必ず DB push を済ませること。

#### 大規模 audit_logs テーブルでの index 作成について

`audit_logs` は append-only で行数が膨らみやすい。本番で `CREATE INDEX` を非 CONCURRENTLY で実行すると **書き込みロック**が走り、全 audit 書き込みが停止する。本番の audit_logs 行数が大きい場合は以下のいずれかで対応:

1. **Drizzle 出力 SQL を手動編集して CONCURRENTLY 化**: drizzle-kit が生成する SQL の `CREATE INDEX` を `CREATE INDEX CONCURRENTLY` に置換してから適用
2. **メンテナンスウィンドウ確保**: 一時的に書き込み停止できる時間帯に通常の `db:push` を流す
3. **Neon の Branch を活用**: 本番ブランチに事前に index を CONCURRENTLY で作成、その後アプリ側の schema 反映

開発環境・新規プロジェクトでは通常の `db:push` で問題なし（テーブル小さい）。
