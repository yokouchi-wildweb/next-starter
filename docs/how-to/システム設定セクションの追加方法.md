# システム設定セクションの追加方法

管理画面の「システム設定」配下に、設定カテゴリを新規追加したり、既存カテゴリを変更する手順。

## 前提

- 設定のスキーマ（Zod）は `src/features/core/setting/setting.extended.ts`
- 設定カテゴリ（管理画面のサブページ）の宣言は `src/features/core/setting/setting.sections.ts`
- 管理メニュー・ページ・フォームはすべて上記セクション定義から **自動生成**

## 基本の追加手順（新規カテゴリ）

### 1. スキーマを追加

`setting.extended.ts` に Zod フィールドを追記:

```ts
export const settingExtendedSchema = z.object({
  // 既存
  maintenanceEnabled: z.coerce.boolean().default(false),
  maintenanceStartAt: z.coerce.date().nullable().default(null),
  maintenanceEndAt: z.coerce.date().nullable().default(null),

  // 追加
  siteTitle: z.string().default(""),
  siteDescription: z.string().default(""),
  ogImageUrl: z.string().default(""),
});
```

DBマイグレーション不要。既存レコードは `.default()` で自動補完される。

### 2. セクションを追加

`setting.sections.ts` にエントリを追加:

```ts
import { Search } from "lucide-react";

export const settingSections: SettingSectionMap = {
  // 既存セクション …

  seo: {
    label: "SEO 設定",
    order: 30,
    icon: Search,
    fields: [
      { name: "siteTitle",       label: "サイトタイトル",   formInput: "textInput" },
      { name: "siteDescription", label: "ディスクリプション", formInput: "textarea" },
      { name: "ogImageUrl",      label: "OG 画像 URL",    formInput: "textInput" },
    ],
  },
};
```

これで完了。以下が自動で動作する:

| 項目 | 結果 |
|---|---|
| URL | `/admin/settings/seo` |
| PageTitle | `label` の値（SEO 設定） |
| サイドバーメニュー | 「システム設定」配下に追加 |
| 入力フォーム | `FieldRenderer` が `formInput` から生成 |
| 保存処理 | 既存の `useUpdateSetting` で共通化 |

## 応用

### 権限でアクセス制御

```ts
security: {
  label: "セキュリティ",
  order: 50,
  allowRoles: ["super_admin"],   // このロールのみメニュー表示 + ページアクセス可
  fields: [ ... ],
},
```

### フィールドのグループ化

```ts
seo: {
  label: "SEO 設定",
  order: 30,
  fields: [ /* 全フィールド */ ],
  fieldGroups: [
    { key: "basic",   label: "基本",    fields: ["siteTitle", "siteDescription"] },
    { key: "social",  label: "SNS共有", fields: ["ogImageUrl"], collapsible: true },
  ],
},
```

### 横並びフィールド

```ts
inlineGroups: [
  {
    key: "dateRange",
    label: "期間",
    fields: ["startAt", "endAt"],
    fieldWidths: ["w-40", "w-40"],
  },
],
```

### カスタム UI

**ファイル拡張子は `.tsx`**（JSX が書ける）。`SettingSection` は FieldRenderer の拡張点を全て公開している。

#### UI を挿入（最も手軽）

```tsx
import { MyBanner } from "@/features/xxx/components/MyBanner";

seo: {
  label: "SEO 設定",
  order: 30,
  fields: [ /* ... */ ],
  beforeAll: <MyBanner />,
  beforeField: {
    siteTitle: <Para tone="muted">SEOの基礎となる重要項目です</Para>,
  },
  afterField: {
    ogImageUrl: <OgImagePreview />,
  },
},
```

#### フィールドUI を完全に自作（`formInput: "custom"`）

`FieldRenderer` は `formInput: "custom"` のフィールドには何も描画しない。`beforeField` / `afterField` 経由で独自コンポーネントを挿入する。コンポーネント側は `Controller` で値と連携する:

```tsx
import { Controller } from "react-hook-form";

const ColorPickerField = () => (
  <Controller
    name="accentColor"
    render={({ field }) => <MyColorPicker value={field.value} onChange={field.onChange} />}
  />
);

branding: {
  label: "ブランディング",
  order: 40,
  fields: [
    { name: "accentColor", label: "アクセントカラー", formInput: "custom" },
  ],
  beforeField: {
    accentColor: <ColorPickerField />,
  },
},
```

#### その他の差し込み口

| セクションプロパティ | 相当する FieldRenderer props | 用途 |
|---|---|---|
| `beforeAll` / `afterAll` | 同名 | フォーム全体の前後にUI |
| `beforeField` / `afterField` | 同名 | 特定フィールドの前後にUI |
| `fieldPatches` | 同名 | 既存フィールドの部分上書き |
| `insertBefore` / `insertAfter` | 同名 | 新規フィールドを特定位置に挿入 |

### 専用ページの独自実装

セクション定義で賄えないほど複雑なUIが必要な場合は、`setting.sections.tsx` に登録せず `app/admin/(protected)/settings/my-custom/page.tsx` を独自に作成してもよい。動的ルートと共存可能（メニュー追加は手動）。

## 使える入力タイプ一覧

`formInput` に指定できる値:

```
textInput, numberInput, textarea, select, multiSelect, combobox,
radio, checkbox, stepperInput, switchInput, dateInput, timeInput,
datetimeInput, emailInput, passwordInput, colorInput, mediaUploader,
hidden, none, custom, asyncCombobox, asyncMultiSelect
```

詳細は `src/components/Form/Field/types.ts` の `FormInputType`。

## 整合性チェック

セクション定義のフィールド名（`field.name`）が Zod スキーマに存在しない場合、起動時に以下のエラーが throw される:

```
[setting.sections] Section "seo" references unknown field "siteTitl". Add it to SettingBaseSchema or settingExtendedSchema first.
```

→ まず Zod スキーマ側に該当フィールドを追加する。

## 旧構造からの移行（既存プロジェクト向け）

旧: `EditSettingForm.tsx` や独自 `AdminSettingEdit` コンポーネントで設定 UI を構築していた。
新: セクション駆動で `setting.sections.ts` に宣言するだけ。

### 移行パターン別対応

#### パターン A: `setting.extended.ts` に項目を追加していたが、UI には出していなかった

- 旧コードでは extended フィールドを追加しても、デフォルトでは UI に出ていなかった
- **対応**: 表示したいフィールドを `setting.sections.ts` の適切なセクションに追加する。必要なら新規セクションを作る

#### パターン B: `EditSettingForm.tsx` を改変して独自の UI を組んでいた

- **対応**: 改変していた内容を FieldConfig 配列に変換して `setting.sections.ts` に移植
- 条件付き表示や複雑な依存があった場合は `formInput: "custom"` で独自コンポーネントを渡す、あるいは専用ページを作って動的ルートと共存

#### パターン C: 独自の `/admin/settings/xxx/page.tsx` を作っていた

- **対応**: `setting.sections.ts` にセクションを登録すれば、自動生成ページで代替できるケースが多い
- 自動生成で賄えないケース（凝った UI、特殊なデータフロー）はそのまま残し、動的ルートと共存してよい
- 共存する場合は、メニューでの URL が被らないように注意

### チェックリスト

- [ ] `setting.extended.ts` の Zod 定義が最新か
- [ ] 旧 UI で表示していたフィールドが全て `setting.sections.ts` に登録されている
- [ ] `/admin/settings/` 配下に旧ページファイル (`general/page.tsx` 等) が残っていない
- [ ] メニューが意図通りに並んでいる（`order` と `icon`）
- [ ] `allowRoles` を指定したセクションが正しいロールのみに見えている
- [ ] 保存→再読込で値が保持されている
