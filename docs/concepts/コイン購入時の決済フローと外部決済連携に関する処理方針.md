# コイン購入時の決済フローと外部決済連携に関する処理方針

アプリ内通貨（コイン、ポイント等）の購入機能における決済フロー、状態管理、二重購入防止の設計をまとめる。

> **関連ドキュメント**:
> - ウォレットの基本設計: [ウォレット機能の設計方針と運用ガイド](./ウォレット機能の設計方針と運用ガイド.md)
> - 自社受付の銀行振込（inhouse プロバイダ）は外部 Webhook を持たない自己申告ベースの特殊フローのため、本ドキュメントとは別に [src/features/core/bankTransferReview/README.md](../../src/features/core/bankTransferReview/README.md) を参照。

## 1. 購入リクエストの管理（purchase_requests）

### 背景と目的

リダイレクト型決済（Square、Fincode 等）では、ユーザーが外部決済ページに遷移し、支払い完了後に戻ってくるまで「処理中」の状態を管理する必要がある。

`wallet_history` は「完了した残高変更」を記録するテーブルであり、以下の中間状態は表現できない：

- 決済ページに遷移中（ユーザーがまだ支払っていない）
- 支払い処理中（決済サービスが処理中）
- 支払い失敗（残高変更は発生しない）

そのため、購入リクエストの状態を管理する専用テーブル `purchase_requests` を導入している。

### purchase_requests テーブル構造

- `id` (UUID) — リクエストID（主キー）
- `user_id` (UUID) — 購入ユーザー
- `wallet_history_id` (UUID, nullable) — 完了時に作成された wallet_history のID
- `idempotency_key` (UUID, UNIQUE) — 冪等キー（二重購入防止）
- `wallet_type` (ENUM) — 対象ウォレット種別（`regular_coin` 等）
- `amount` (INTEGER) — 購入数量（コイン数など）
- `payment_amount` (INTEGER) — 支払い金額（円）
- `payment_method` (TEXT) — 支払い方法（`credit_card` / `convenience_store` 等。値は `payment.config.ts` の `paymentMethods[i].id` と一致する）。初期値はユーザーが UI で選択した値で、Webhook 受信時にプロバイダから取得した実際の決済方法で上書きされる
- `status` (ENUM) — `pending` / `processing` / `completed` / `failed` / `expired`
- `payment_provider` (TEXT) — 決済プロバイダ（`dummy` / `square` / `fincode` / `stripe` 等）。`paymentMethods[i].provider` から `resolveProviderForMethod()` で解決された値
- `payment_session_id` (TEXT, nullable) — 外部決済サービスのセッションID
- `provider_order_id` (TEXT, nullable) — プロバイダ側の注文ID（Fincode 等で使用）
- `transaction_id` (TEXT, nullable) — プロバイダ側のトランザクションID
- `redirect_url` (TEXT, nullable) — 決済ページへのリダイレクトURL
- `error_code` (TEXT, nullable) — エラー時のコード
- `error_message` (TEXT, nullable) — エラー時の詳細メッセージ
- `webhook_signature` (TEXT, nullable) — Webhook 署名検証用
- `coupon_code` (TEXT, nullable) — 適用されたクーポンコード
- `discount_amount` (INTEGER, nullable) — クーポンによる割引額
- `original_payment_amount` (INTEGER, nullable) — 割引前の支払い金額
- `coupon_redeem_failed` (BOOLEAN, nullable) — クーポン消化失敗フラグ
- `milestone_results` (JSONB, nullable) — マイルストーン評価結果
- `completed_at` (TIMESTAMP, nullable) — 完了日時
- `paid_at` (TIMESTAMP, nullable) — 決済完了日時
- `expires_at` (TIMESTAMP, nullable) — 有効期限
- `created_at` (TIMESTAMP) — 作成日時
- `updated_at` (TIMESTAMP) — 更新日時

### ステータス遷移

```
┌─────────┐
│ pending │ ← 購入開始、決済セッション作成前
└────┬────┘
     │ 決済セッション作成成功
     ▼
┌────────────┐
│ processing │ ← ユーザーが決済ページで支払い中
└─────┬──────┘
      │
      ├── Webhook: 成功 ─────────→ ┌───────────┐
      │                             │ completed │ → wallet残高更新
      │                             └───────────┘
      │                                   ▲
      ├── Webhook: 失敗 ─────────→ ┌──────┴─┐
      │                             │ failed │ ── Webhook順序逆転の救済 ──→ completed
      │                             └────────┘    （PENDING→failed 後に COMPLETED が来た場合）
      │
      └── バッチ処理 ────────────→ ┌─────────┐
          (expires_at < now)       │ expired │
                                   └─────────┘

注: failed/expired からの initiate は 400 エラー（新しい購入を開始してもらう）
```

### wallet / wallet_history との関係

```
purchase_requests (1) ──→ (0..1) wallet_history
        │
        │ 完了時のみ履歴が作成される
        │
        └──→ wallet_history.request_batch_id = purchase_requests.id
             として紐付け
```

- `purchase_requests` は「購入の意図と状態」を管理
- `wallet_history` は「実際に発生した残高変更」を記録
- 両者は独立しており、購入が完了した場合のみ `wallet_history` にレコードが作成される

## 2. ファイル構成

```
src/features/core/purchaseRequest/services/server/wrappers/
  purchaseService.ts     — 型定義 + re-export ハブ + getPurchaseStatus/ForUser
  initiatePurchase.ts    — 購入開始（パッケージ照合、クーポン検証、セッション作成）
  completePurchase.ts    — 購入完了（楽観的ロック、ウォレット更新、マイルストーン）
  failPurchase.ts        — 購入失敗（楽観的ロック）
  webhookHandler.ts      — Webhook処理（pending分岐、エラーハンドリング）
  purchaseHelpers.ts     — findByWebhookIdentifier、findByIdempotencyKey、
                           handleExistingRequest、expirePendingRequests
```

各ファイルの責務:

- **purchaseService.ts**: 全関数の re-export ハブ。型定義（`InitiatePurchaseParams` 等）と `getPurchaseStatus` / `getPurchaseStatusForUser` を含む
- **initiatePurchase.ts**: 購入開始の全ロジック。冪等キーチェック、パッケージ照合、クーポン検証、セッション作成、ステータス更新まで一貫して処理
- **completePurchase.ts**: Webhook 成功時またはポーリング検知時の完了処理。楽観的ロック、ウォレット残高更新、クーポン消化、ポストフック、マイルストーン評価
- **failPurchase.ts**: Webhook 失敗時の処理。楽観的ロックで completed を上書きしないことを保証
- **webhookHandler.ts**: Webhook エントリポイント。プロバイダの verifyWebhook を呼び出し、結果に応じて completePurchase / failPurchase に振り分け
- **purchaseHelpers.ts**: 共通ヘルパー。Webhook 識別子による検索、冪等キーによる検索、既存リクエストのハンドリング、期限切れバッチ処理

## 3. 決済フローの設計

### リダイレクト型決済の全体フロー

```
┌──────────────────────────────────────────────────────────────────────────┐
│ 1. ユーザーが購入ボタンを押す（支払い方法選択済み）                          │
│    └→ POST /api/wallet/purchase/initiate                                 │
│       ├ paymentMethod の妥当性チェック（available + provider enabled）     │
│       ├ resolveProviderForMethod(paymentMethod) で担当プロバイダを解決      │
│       ├ idempotency_key で既存チェック                                    │
│       │  ├ pending → 再利用（クーポン情報を更新）                          │
│       │  └ processing 以降 → handleExistingRequest で既存を返す            │
│       ├ パッケージ照合（CURRENCY_CONFIG との一致検証）                      │
│       ├ クーポン検証（validateForCategory）                               │
│       ├ 割引後金額の 0 以下チェック                                       │
│       ├ purchase_request 作成（status: pending, payment_method=選択値）   │
│       └ 解決されたプロバイダにセッション作成リクエスト                       │
├──────────────────────────────────────────────────────────────────────────┤
│ 2. 決済セッション作成完了                                                  │
│    ├→ provider_order_id を保存（Fincode の場合）                          │
│    ├→ purchase_request 更新                                              │
│    │   status: processing, payment_session_id, redirect_url              │
│    └→ クライアントに決済ページ URL を返却                                  │
├──────────────────────────────────────────────────────────────────────────┤
│ 3. ユーザーが決済ページで支払い                                            │
│    └→ 外部決済サービスのページにリダイレクト                                │
├──────────────────────────────────────────────────────────────────────────┤
│ 4. 支払い完了後の並行処理                                                  │
│                                                                          │
│    [Webhook 経路] ────────────────────────────────────────────────────   │
│    │  POST /api/webhook/payment                                          │
│    │  ├ プロバイダの verifyWebhook で署名検証 + 結果パース                 │
│    │  ├ PaymentResult.status が "pending" → スキップ（200 を返す）         │
│    │  ├ success → completePurchase（残高更新、クーポン消化、マイルストーン） │
│    │  └ failure → failPurchase                                           │
│    │                                                                      │
│    [ユーザー経路] ───────────────────────────────────────────────────    │
│    │  リダイレクト: /api/wallet/purchase/callback                         │
│    │  └ purchase_request の status を確認                                 │
│    │     ├ completed → /wallet/[slug]/purchase/complete へリダイレクト     │
│    │     ├ processing → ポーリングで完了を待機                             │
│    │     └ failed/expired → /wallet/[slug]/purchase/failed へリダイレクト  │
│    │                                                                      │
│    [ポーリング経路] ─────────────────────────────────────────────────    │
│    │  クライアントが 2 秒間隔で getPurchaseStatusForUser を呼び出し        │
│    │  └ サーバー側でプロバイダに問い合わせ                                  │
│    │     ├ completed 検知 → completePurchase を呼び出し                    │
│    │     └ failed 検知 → failPurchase を呼び出し                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### ページ構成（ウォレット通貨ごと）

- `/wallet/[slug]` — 残高表示・購入数量選択
- `/wallet/[slug]/purchase?amount=N&price=N` — 購入確認ページ（クーポン入力・支払い方法選択・購入実行）
- `/api/wallet/purchase/callback` — コールバック（リダイレクト振り分け）
- `/wallet/[slug]/purchase/complete` — 完了ページ
- `/wallet/[slug]/purchase/failed` — 失敗ページ

## 4. 購入開始フロー（initiatePurchase）の詳細

```typescript
async function initiatePurchase(params: InitiatePurchaseParams) {
  // 1. paymentMethod の妥当性チェック
  //    isPaymentMethodSelectable(paymentMethod) === false → 400 エラー
  //    （status="available" 以外、provider 未設定、provider が disabled の場合に false）

  // 2. プロバイダ解決（優先順位）
  //    a. params.paymentProvider が明示指定されていればそれを使用
  //    b. resolveProviderForMethod(paymentMethod) → paymentMethods[i].provider から解決
  //    isPaymentMethodSelectable を通過したメソッドに対して b は必ず成功する。
  //    解決失敗は config 整合性異常として 500 で fail-fast。

  // 3. 冪等キーで既存チェック
  const existing = await findByIdempotencyKey(params.idempotencyKey);
  if (existing) {
    if (existing.status === "pending") {
      // pending なら再利用してクーポン情報を更新
      // → セッション作成に進む
    } else {
      // processing 以降は handleExistingRequest で既存を返す
      return handleExistingRequest(existing);
    }
  }

  // 4. パッケージ照合
  //    CURRENCY_CONFIG のパッケージと amount / paymentAmount の一致を検証
  //    → 改竄されたリクエストを拒否

  // 5. クーポン検証（validateForCategory）
  //    → 無効なクーポンは適用せず続行 or エラー

  // 6. 割引後金額の 0 以下チェック
  //    actualPaymentAmount <= 0 → 400 エラー

  // 7. purchase_request 作成（新規 or pending 再利用でクーポン更新）
  //    status: pending, payment_method=ユーザー選択値,
  //    coupon_code, discount_amount, original_payment_amount

  // 8. 決済プロバイダでセッション作成（paymentMethod を渡す）
  //    プロバイダは methodMapping で API 固有 ID に変換して使用
  const session = await provider.createSession(sessionParams);

  // 9. provider_order_id を保存（Fincode の場合）

  // 10. status を processing に更新
  //     payment_session_id, redirect_url を記録

  return { redirectUrl: session.redirectUrl, purchaseRequestId };
}
```

## 5. Webhook 処理フロー

### webhookHandler（エントリポイント）

```typescript
async function handleWebhook(request: Request, provider: string) {
  // 1. プロバイダの verifyWebhook を呼び出し（署名検証 + 結果パース）
  const result: PaymentResult = await paymentProvider.verifyWebhook(request);

  // 2. PaymentResult.status が "pending" → スキップ（200 を返す）
  if (result.status === "pending") return;

  // 3. success → completePurchase
  if (result.success) {
    try {
      await completePurchase(result);
    } catch (e) {
      if (e instanceof DomainError) return; // 200 で返す（リトライ防止）
      throw e; // 500 系は re-throw（プロバイダにリトライさせる）
    }
  }

  // 4. failure → failPurchase（同様の try-catch パターン）
  if (!result.success) {
    try {
      await failPurchase(result);
    } catch (e) {
      if (e instanceof DomainError) return;
      throw e;
    }
  }
}
```

### Webhook route（route.ts）のエラーハンドリング

- 署名検証失敗 → **401**（正当な拒否）
- handleWebhook の DomainError → **200**（リトライ防止）
- handleWebhook の 500 系エラー → **500**（プロバイダにリトライさせる）
- 404（セッション不明） → **200**（リトライ防止）

### Square プロバイダの verifyWebhook 3 分岐

- `COMPLETED` / `APPROVED` → `{ success: true, status: "completed" }`
- `CANCELED` / `FAILED` → `{ success: false, status: "failed" }`
- それ以外（`PENDING` 等） → `{ success: false, status: "pending" }`

## 6. completePurchase の詳細

```typescript
async function completePurchase(result: PaymentResult) {
  // 1. findByWebhookIdentifier で検索
  //    payment_session_id → UUID → プロバイダ固有リゾルバー
  const purchaseRequest = await findByWebhookIdentifier(result);

  // 2. completed なら冪等に早期リターン
  if (purchaseRequest.status === "completed") return;

  // 3. processing または failed のみ許可（それ以外は 400）
  //    failed → completed の遷移は Webhook 順序逆転の救済

  // 4. paidAmount がある場合、DB の payment_amount と照合（改竄検知）
  //    → 不一致時は markAsFatalFailure で status を failed に遷移してから throw
  //    → これがないと webhookHandler が DomainError を 200 で飲み込む結果、
  //      リクエストが processing のまま塩漬けになり、ポーリング側が無限スピナーになる

  // 5. トランザクション内で以下を実行:
  await db.transaction(async (tx) => {
    // 5a. 楽観的ロック
    //     WHERE id = ? AND status IN ('processing', 'failed')
    //     → 0 行更新なら競合（既に別の処理で完了済み）

    // 5b. adjustBalance でウォレット残高更新
    //     requestBatchId = purchaseRequest.id

    // 5c. wallet_history_id を記録

    // 5d. クーポン redeem
    //     失敗時は coupon_redeem_failed = true に設定
    //     → 購入自体はブロックしない（クーポン消化失敗は許容）

    // 5e. ポストフック実行（SAVEPOINT で隔離）
    //     → ポストフック失敗が購入完了を巻き戻さない

    // 5f. マイルストーン評価・永続化
    //     → milestone_results に保存
  });
}
```

---

### completePurchase 内で「修復不能エラー」を投げる場合の規約

`webhookHandler` は `completePurchase` が投げた `DomainError` を、決済プロバイダの不要なリトライを防ぐために 200 として飲み込む。このため、**ステータス遷移なしに DomainError を投げると、リクエストが永久に `processing` のまま塩漬けになる**（クライアント側のポーリングは検知できず、60秒のタイムアウトを待つ）。

これを防ぐため、修復不能なエラー（金額不一致など、リトライしても解決しない種類のエラー）を投げる前には、必ず内部ヘルパー `markAsFatalFailure(requestId, errorCode, errorMessage)` を呼び出して `status` を `failed` に遷移させること。

```typescript
if (paidAmount != null && paidAmount !== purchaseRequest.payment_amount) {
  console.error("[completePurchase] 決済金額不一致: ...");
  await markAsFatalFailure(
    purchaseRequest.id,
    "AMOUNT_MISMATCH",
    "決済金額がリクエストの金額と一致しません。",
  );
  throw new DomainError("決済金額がリクエストの金額と一致しません。", { status: 400 });
}
```

`markAsFatalFailure` は `WHERE status = 'processing'` の楽観ロック付きなので、すでに `completed` や `failed` になっているレコードを上書きすることはなく、既存の冪等性を破壊しない。将来 `completePurchase` 内に新しい修復不能エラー分岐を追加する場合は、この規約に従うこと。

なお、以下のケースでは `markAsFatalFailure` は呼ばない:
- 「すでに completed」→ ステップ2の早期リターンで処理済み
- 「楽観ロック競合 (409)」→ 別プロセスがすでに正しく処理しているため
- 「無効なステータス (`pending`/`expired` 等)」→ そもそも `processing` ではないので遷移しない方が安全

async function failPurchase(result: PaymentResult) {

  const purchaseRequest = await findByWebhookIdentifier(result);

  // 3. 楽観的ロック

## 8. Callback と Webhook の役割分担

Callback ページと Webhook は補完関係にあり、両方が必要。

- **Webhook**: 確実な決済完了の記録（サーバー間通信 + リトライ機構、署名検証、残高更新の実行）
- **Callback**: ユーザーへの即時フィードバック（Webhook との競合状態の吸収、状態に応じたページ振り分け）

**重要**: 実際のコイン付与（残高更新）は Webhook またはポーリング経由の completePurchase で行う。Callback ページ自体は DB の状態を確認して表示・リダイレクトするだけ。

### クライアント側ポーリング（usePurchaseStatusPolling）

- **間隔**: 2 秒ごと
- **上限**: 最大 30 回（60 秒）
- **仕組み**: `getPurchaseStatusForUser` がサーバー側でプロバイダに問い合わせ
- **検知時の動作**: completed / failed 検知時に `completePurchase` / `failPurchase` を呼び出し
- **タイムアウト時**: エラー表示（サポート案内）

## 9. 二重購入防止と冪等性（多層防御）

### 二重購入が発生するパターンと対策

- ダブルクリック → UI（ボタン無効化）
- リロード → API（冪等キー）
- 戻るボタン → API（冪等キー）
- ネットワーク遅延 → API（冪等キー）
- Webhook 重複 → DB（楽観的ロック）
- 複数タブ → DB（UNIQUE 制約）

### 多層防御の実装

#### レイヤー 1: UI（補助的）

```typescript
// useMemo で idempotencyKey をマウント時 1 回生成
const idempotencyKey = useMemo(() => crypto.randomUUID(), []);

// isLoading でボタン無効化（連打防止）
<Button disabled={isLoading} onClick={handlePurchase}>
  {isLoading ? "処理中..." : "購入する"}
</Button>
```

#### レイヤー 2: API（冪等キー）

```typescript
// サーバー側: 同じキーのリクエストは既存を返す
const existing = await findByIdempotencyKey(idempotencyKey);
if (existing) {
  if (existing.status === "pending") {
    // pending なら再利用してクーポン情報を更新
  } else {
    // processing 以降は handleExistingRequest で既存の結果を返す
    return handleExistingRequest(existing);
  }
}
```

#### レイヤー 3: DB（UNIQUE 制約）

```sql
-- idempotency_key に UNIQUE 制約
CREATE UNIQUE INDEX purchase_requests_idempotency_key_idx
  ON purchase_requests(idempotency_key);
```

#### レイヤー 4: Webhook（楽観的ロック）

```sql
-- ステータス更新時の楽観的ロック
-- WHERE status IN (...) で 1 回だけ UPDATE 成功
UPDATE purchase_requests
SET status = 'completed', completed_at = NOW()
WHERE id = $1 AND status IN ('processing', 'failed')
RETURNING *;
-- 0 行更新 = 既に別の処理で更新済み
```

#### レイヤー 5: Webhook エラーハンドリング

```typescript
// handleWebhook 内で DomainError を catch して 200 を返す
// → プロバイダのリトライを防止（同じ処理が再実行されない）
try {
  await completePurchase(result);
} catch (e) {
  if (e instanceof DomainError) return; // 200 で返す
  throw e; // 500 系のみリトライ対象
}
```

## 10. セキュリティ対策

### リクエスト改竄防止

- **パッケージ照合**: クライアントから送信された `amount` / `paymentAmount` が `CURRENCY_CONFIG` に定義されたパッケージと一致するか検証。存在しない金額の組み合わせを拒否
- **支払い方法のホワイトリスト検証**: API スキーマで `paymentMethod` を `isPaymentMethodSelectable` で `.refine` 制約。`status="available"` かつ provider が enabled のメソッドのみ受付（`coming_soon` / `disabled` の悪用防止）
- **割引後金額チェック**: `actualPaymentAmount <= 0` で拒否（100% 割引クーポンの悪用防止）

### Webhook の信頼性

- **金額照合**: Webhook から通知された `paidAmount` と DB の `payment_amount` の一致を検証（決済金額の不整合検知）
- **Fincode 署名検証**: `timingSafeEqual` で比較（タイミング攻撃防止）
- **Square 署名検証**: HMAC-SHA256 + `timingSafeEqual`

### インフラレベル

- **Host Header Injection 防止**: リダイレクト URL 生成時に `businessConfig.url` を使用（リクエストヘッダーの Host に依存しない）
- **Dummy プロバイダ**: `NODE_ENV=production` でブロック（本番環境でのテスト用プロバイダ利用を防止）
- **PII ログ制御**: `paymentConfig.debugLog` フラグで切替（個人情報を含むデバッグログの出力を制御）

## 11. クーポン連携

### 購入開始時（initiatePurchase）

- `validateForCategory` でクーポンの有効性を検証
- 有効な場合、`discount_amount`、`original_payment_amount`、`coupon_code` を purchase_request に記録
- pending 再利用時はクーポン情報を更新（ユーザーがクーポンを変更した場合に対応）

### 購入完了時（completePurchase）

- トランザクション内でクーポンを redeem（消化）
- **消化失敗は購入をブロックしない**: 失敗時は `coupon_redeem_failed = true` に設定
- 理由: 決済は既に成功しており、クーポンの消化失敗で購入を取り消すべきではない

## 12. マイルストーン連携

### completePurchase 内での処理

- トランザクション内でマイルストーンを評価
- 結果は `milestone_results` カラムに JSONB で永続化
- 完了ページでマイルストーン達成通知を表示するために使用

## 13. 期限切れ処理

- `purchase_requests.expires_at` で有効期限を管理
- バッチジョブで `expires_at < now` かつ `status = 'pending'` のレコードを `expired` に更新
- `expirePendingRequests` ヘルパーで実行
