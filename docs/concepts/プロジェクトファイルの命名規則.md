# 📄 プロジェクトファイルの命名規則

Next.js（App Router）ベースで運用する本プロジェクトにおける命名ルールを以下にまとめます。層ごとの責務を保ちながら、誰が見ても構造と役割が理解できる名前付けを徹底してください。

---

## 命名ルール一覧

| 種別                                                | 命名規則 | 補足 |
|---------------------------------------------------|-----------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| ルート定義 (`app/**`)                                  | Next.js が規定する固定ファイル名を使用（例: `page.tsx`, `layout.tsx`, `template.tsx`, `loading.tsx`, `route.ts`） | ルートセグメントのディレクトリはケバブケース。グルーピングには `(admin)` のように括弧を利用する。 |
| ルートディレクトリ                                         | ケバブケース。グループ化は括弧（例: `(marketing)`） | ルート階層に命名の揺らぎを持ち込まないため、単語の区切りはすべてハイフンで統一する。 |
| コンポーネント名                                          | パスカルケース（例: `UserMyPage`） | `index.tsx` で公開するセクションコンテナのエクスポート名もパスカルケースで統一する。 |
| コンポーネントファイル                                       | パスカルケース、もしくは固定の `index.tsx`（例: `Header.tsx`, `index.tsx`）※ Shadcn 関連など一部例外あり | `index.tsx` を入口にし、サブコンポーネントは同ディレクトリ内でパスカルケース命名。Shadcn など外部生成物は元の命名を保持してよい。 |
| コンポーネントディレクトリ                                     | パスカルケース | `src/features/<domain>/components/` 配下のディレクトリ名もパスカルケースで揃える。 |
| フック名                                              | `use` から始まるキャメルケース（例: `useCreateTitle`） | - |
| フックファイル                                           | メインエクスポートのフック名と統一する（例: `useCreateTitle.ts`） | - |
| フックディレクトリ                                         | 小文字単ワード（推奨）もしくはキャメルケース | 例: `src/features/title/hooks/useCreateTitle.ts`。用途が明確な場合のみ複合語を許容する。 |
| サービス名                                             | キャメルケース（例: `titleClient`） | - |
| サービスファイル                                          | メインエクスポートのサービス名と統一する（例: `titleClient.ts`） | ファイル名は役割名＋ドメインで揃える。 |
| サービスディレクトリ                                        | 小文字単ワード（推奨）もしくはキャメルケース | `services/client/` と `services/server/` の階層名を固定で使用する。 |
| ユーティリティ関数名                                        | キャメルケース（例: `zodHelpers`） | - |
| ユーティリティ関数ファイル                                     | キャメルケース（例: `array.ts`, `siteTheme.ts`） | - |
| ユーティリティ関数ディレクトリ                                   | 小文字単ワード（推奨）もしくはキャメルケース | - |
| ライブラリファイル (`src/lib/**`)                          | 原則キャメルケース | 外部ライブラリ由来のファイルはプロジェクト規則へ統一しなくてもよいが、ラップする際は自前の命名規則に従う。 |
| ライブラリディレクトリ                                       | 原則キャメルケース | 外部コードをそのまま配置する場合のみ命名揺らぎを許容する。 |
| エンティティ名<br/>(`src/features/<domain>/entities/**`) | パスカルケース（例: `Title`, `TitleSchema`, `SampleTable`） | `schema.ts` や `drizzle.ts` 内で定義する const（テーブル、列挙、リレーションテーブルなど）もパスカルケースで統一する。 |
| エンティティファイル                                        | 小文字単ワード（推奨）もしくはキャメルケース | 例: `schema.ts`, `drizzle.ts`, `form.ts`。 |
| エンティティディレクトリ                                      | ディレクトリ分けは非推奨 | - |
| 定数名                                               | 大文字スネークケース（例: `USER_PROVIDER_TYPES`） | - |
| 定数ファイル                                            | 小文字単ワード（推奨）もしくはキャメルケース | - |
| 定数ディレクトリ                                          | ディレクトリ分けは非推奨 | - |
| タイプ名                                              | パスカルケース（例: `FieldType`） | - |
| タイプファイル                                           | 小文字単ワード（推奨）もしくはキャメルケース | - |
| タイプディレクトリ                                         | ディレクトリ分けは非推奨 | - |


> 💡 `Shadcn` が生成する UI や `src/lib/**` の一部など、外部ライブラリ由来のファイルは生成時の命名を維持して問題ありません。ラップする場合のみプロジェクトの命名規則へ揃えてください。
