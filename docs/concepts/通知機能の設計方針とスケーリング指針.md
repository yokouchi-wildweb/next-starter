# 通知機能の設計方針とスケーリング指針

通知（notification）ドメインの設計思想・計算量特性・大規模化したときの拡張判断基準をまとめる。
「いま何が効いていて、負荷が出たら次に何を入れるか」を downstream フォークが判断できることを目的とする。

対象コード: `src/features/core/notification/`

---

## 1. 基本設計：query-time fan-out（配信レコードを持たない）

通知は **配信時にユーザーごとのレコードを作らず、読み取り時にクエリで対象者を動的に判定する**設計を採る。

- 通知1件 = `notifications` テーブル1行（`target_type` = all / role / individual と対象情報のみ保持）
- 「ユーザー X にこの通知が見えるか」は読み取り時に WHERE 条件で計算する
- 既読は別管理（後述のハイブリッド方式）

この設計の最大の利点は **配信（書き込み）が O(1)** であること。`target_type='all'` で全ユーザーに配信しても **1行 INSERT** で済み、ユーザー数に依存しない。

トレードオフとして、コストは読み取り側に寄る。読み取り側のスケール対策が本ドキュメントの中心である。

### この設計が得意 / 不得意な配信パターン

- **得意**: all / role 中心のブロードキャスト。本テンプレートの想定主用途
- **不得意**: 巨大なユーザー集合への `individual` 配信（`target_user_ids` が単一行の巨大配列になる）、動的な任意セグメント配信。これらが主用途になる場合は §6 の recipient モデルを検討する

---

## 2. 可視性ルールの単一集約点

「誰にどの通知が見えるか」のルールは `services/server/notification/queryHelpers.ts` の
`buildVisibilityWhere(viewer)` 1箇所に集約されている。ユーザー向けクエリ
（一覧 / ページ / 件数 / 全既読）はすべてこれを通る。

現在のルール:

- `published_at <= now()`（公開済み。予約投稿は時刻到来まで非表示）
- `published_at >= 閲覧者の登録日時`（登録日以前の通知は閲覧不可。新規ユーザーに過去の全体配信が見えてしまう問題への対策。`created_at` が NULL のときは fail-closed）
- ターゲット一致（all / role 一致 / userId 一致）

### 拡張点：NotificationViewer

可視性判定に必要な閲覧者情報は `services/server/notification/viewer.ts` の `NotificationViewer` 型に集約されている。ルールにユーザー属性を増やすときは「Viewer に属性追加 → `buildVisibilityWhere` に WHERE 句追加」の2箇所だけで全消費者へ波及する（呼び出しシグネチャは変えない）。

---

## 3. 既読のハイブリッド方式

既読は2系統を併用し、判定ロジックは `services/server/notification/readState.ts` の1モジュールに集約されている（クエリ各所はここを通すだけで、どちらの系統で既読かを意識しない）。

1. **個別既読** … `notification_reads`（通知1件ごとの行）。通知ごとの read/unread を表現
2. **既読ウォーターマーク** … `notification_read_states.read_watermark_at`。この時刻以前に公開された通知は既読扱い。「全既読」を O(1) で表現

「未読」の単一定義（`unreadConditions`）= 個別既読が無い AND サイレントでない AND ウォーターマーク以降に公開。
既読セマンティクスを変更するときは `readState.ts` だけを変更する。

### 全既読は O(1)

`markAllAsRead` は未読を1件ずつ INSERT する代わりに、watermark を `now()` に upsert（1行）。
あわせて同一トランザクションで watermark 以前の個別既読行を prune し、`notification_reads` の増殖を抑制する。

### 注意：backdate された通知

published_at を過去日時にして配信した通知が、既存ユーザーの watermark より古い時刻だと「最初から既読」として扱われる。通常運用（published_at = 現在 or 予約時刻）では問題ないが、意図的に過去日時を入れる場合は留意する。

## 4. 読み取りの計算量とインデックス

| 操作 | 方式 | 計算量 |
|------|------|--------|
| 配信（all/role/individual） | 1行 INSERT | O(1) |
| 一覧ページ取得（無限スクロール） | keyset(cursor) ページネーション | O(page) |
| 全既読 | watermark upsert + prune | O(1) + prune |
| 未読カウント | watermark 以降を COUNT | O(watermark 以降の件数) |

インデックス: `notifications (published_at, id)` 複合インデックス。

- keyset の行値比較 `(published_at, id) < cursor` と `ORDER BY published_at DESC, id DESC` を索引で処理
- 左端プレフィックスにより `published_at` 単独の範囲フィルタ（可視性判定）も効く

ページ取得は `total` を返さない（keyset では安価に出せないため）。件数が必要なら `/api/notification/my/count` を併用する。

## 5. 現状でカバー済みのスケール対策（無インフラ）

以下は追加インフラ不要で導入済み。all/role ブロードキャスト中心の用途では、これらで読み取りパスのスケール critical な対策はほぼ出し切っている。

1. `notifications (published_at, id)` 複合インデックス
2. 既読ウォーターマーク（全既読 O(1）・既読行の増殖抑制・未読集計対象の縮小）
3. keyset ページネーション（深いスクロールでも O(page)）

---

## 6. これ以上スケールが必要になったときの拡張判断

ここから先は **インフラ追加** または **アーキテクチャ変更** を伴う。実測で必要になってから opt-in で導入することを推奨する（過剰実装を避ける）。

### 未読カウントのキャッシュ化

- **トリガー**: バッジ用 `unread-count` のポーリング QPS が DB を圧迫し始めたとき
- **手段**: 短 TTL キャッシュ（Redis 等）でカウントをキャッシュ。または SWR の `dedupingInterval` / ポーリング間隔を調整して頻度を抑制
- **注意**: per-user カウンタ列を持つ方式は、ブロードキャスト配信のたびに全ユーザー分の fan-out 更新が必要になり、§1 の「配信 O(1)」の利点を失うため不可。短 TTL キャッシュ方式を採る

### リアルタイム配信

- **トリガー**: ポーリングでは体感が不十分（即時通知が要件）になったとき
- **手段**: SSE / WebSocket / Web Push。transport インフラの追加が必要
- プロダクト要件駆動。スケールというより機能要件

### recipient / inbox モデル（query-time fan-out からの脱却）

- **トリガー**: 次のいずれかが主用途になったとき
  - 巨大なユーザー集合への `individual` 配信（`target_user_ids` の配列肥大が問題化）
  - 動的な任意セグメント配信（課金ユーザー、休眠復帰 等）
  - 配信時点の凍結（送信後にユーザーの role が変わっても見え方を固定したい）
  - 多チャネル配信・配信停止・配信分析の作り込み
- **手段**: `notification_recipients (notification_id, user_id, delivered_at, read_at, ...)` を導入し、配信を「計算」から「事実」へ。配信時に対象ユーザー分の行を fan-out（キュー/ジョブが必要）
- **コスト**: 配信が O(対象ユーザー数) になり fan-out インフラが要る。§1 の配信 O(1) の利点を失う大きなアーキテクチャ変更
- **備考**: 可視性判定が `buildVisibilityWhere` / `NotificationViewer` に集約されているため、移行時の影響範囲は比較的限定できる
